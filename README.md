# dsl-bootstrap

`dsl-bootstrap` is a Haskell framework experiment for describing an application as declarative source:

- workflow AST
- effect theory
- runtime handler environment
- semantic evidence generated by the framework report

The active packages are:

```text
new-framework-core
  Kernel, public facade, runtime, proof API, self-domain source, and framework reports.

domain-app
  Frontend/domain source written with the same facade DSL style as framework-core.
```

## Source Style

Framework-core expresses itself through domain modules:

```text
new-framework-core/src/Domain/AppBlueprint.hs
new-framework-core/src/Domain/Effects.hs
new-framework-core/src/Domain/Vocabulary.hs
```

Domain-app uses the same layer shape:

```text
domain-app/src/Domain/AppBlueprint.hs
domain-app/src/Effects/Theory.hs
domain-app/src/Domain/Runtime.hs
domain-app/src/Domain/RegistryCodegenSpec.hs
domain-app/src/Domain/SemanticEvidence.hs
domain-app/app/Main.hs
```

The frontend entrypoint is:

```haskell
main :: IO ()
main =
  currentInterpreter currentAst currentEffects
```

Run it with:

```powershell
stack exec mytest
```

## Public Facade

Frontend/domain modules import the public facade:

```haskell
import Framework.Workflow
import Framework.Effect
import Framework.Background
```

Stable facade modules:

```text
Framework.Workflow
Framework.Effect
Framework.Background
Framework.Domain
Framework.Runtime
Framework.FixedPoint
Framework.RegistryCodegen
Framework.SelfArtifact
```

Boundary rules:

```text
Bootstrap.* has zero Framework.* imports.
domain-app/src has zero Bootstrap.* imports.
new-framework-core/src/Domain/* is the self-domain facade expression for framework-core.
```

## Runtime

`Framework.Background` exposes the production runtime API through `Framework.Runtime`.

Current runtime behavior includes:

- typed runtime values
- typed handler results
- transform registry execution
- middleware enter/exit trace
- callback trigger/completion trace
- suspense request trace
- loop start trace
- fact closure execution
- handler coverage reporting
- runtime failure diagnosis

The domain app demonstrates typed runtime flow:

```text
AskUserName -> UserName
UserNameToReportInput: UserName -> ReportInput
GenerateReport: ReportInput -> ReportOutput
```

Both framework-core and domain-app use this shape through facade imports.

## Effect System

The effect DSL supports:

- fact declarations
- dependencies through `needs`
- pipe input through `take`
- pipe output through `make`
- external sends through `uses`
- external boundaries through `externalMake`
- transform declarations
- error handler dispatch through `error`
- send idempotency and retry policy
- handler coverage in reports

Framework-core self expression uses `Framework.Effect` in `new-framework-core/src/Domain/Effects.hs`.
Domain-app uses the same facade in `domain-app/src/Effects/*`.

## Diagnosis And SMT

`Framework.Background` re-exports:

```text
Framework.Background.RuntimeDiagnosis
Framework.Background.ConstraintProof
```

Runtime diagnosis covers:

- handler failure
- error handler dispatch with `ErrorInput`
- idempotent `RetryOnce` replay
- diagnosis probes
- non-idempotent replay blockers

Constraint/proof support covers:

- constraint IR extraction from an app plan
- pure Haskell proof evidence
- optional `z3` or `cvc5` solver witness
- render helpers for constraint facts, errors, and SMT results

## Semantic Evidence

Semantic checks live in `Framework.Domain` reports.

Built-in evidence:

```text
constraint-ir-built

constraint-negative-check

```

Domain-app registers runtime diagnosis and registry codegen evidence in `domain-app/src/Domain/SemanticEvidence.hs`:

```text

runtime-diagnosis-non-idempotent-blocker
```

`domain-app-report` and `domain-app-self-smoke` consume the same `domainAppDomain` registration, so frontend source, runtime handlers, proof evidence, and diagnosis evidence close through one domain definition.

## Self Artifact Gate

The checked-in framework source is the core source.

`Framework.SelfArtifact` describes the Stage 1 artifact manifest and the commands required to accept it as a replacement candidate. The witness materializes a fresh package tree at:

```text
.generated/stage1-framework
```

Then it runs the Stage 1 package through:

```text
stack build
stack exec bootstrap-report
stack exec fixed-point-smoke
stack exec domain-app-report
stack exec registry-codegen-witness
```

This gate proves that the current framework source can stand up as an isolated next package. Runtime use and domain authoring use the checked-in source directly.

Run it with:

```powershell
stack exec self-artifact-witness
```
